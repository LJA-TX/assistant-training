## Optionality Assessment

### Recommended Blocker-Oriented Core

For future blocker-oriented surfaces, the following steps should be treated as mandatory core steps:

1. blocker entry qualification
   - equivalent to Package `5A`
   - needed to distinguish active blocker work from evidence-absent or doctrinally closed surfaces
2. blocker persistence validation
   - equivalent to Package `5B`
   - needed when blocker evidence is currently bounded or single-run
3. blocker characterization
   - equivalent to Package `5C`
   - needed before any meaningful recommendation about doctrine, implementation, or later lifecycle work

### Conditionally Mandatory Step

The equivalent of Package `5D` should be treated as conditionally mandatory.

It is required when:

1. the blocker could reflect both governance-preserving missingness and implementation-path incompleteness;
2. later scorer-centered or implementation-centered work might be contemplated.

It is likely optional when:

1. the surface is plainly evidence-absent, as with the B1 symbol-membership surface;
2. the surface is doctrinally closed rather than blocker-driven;
3. the blocker characterization already proves a single uncontested cause.

### Optional Versus Mandatory Summary

| Step type | Recommendation |
|---|---|
| blocker entry qualification (`5A`-like) | mandatory for blocker-oriented surfaces |
| blocker persistence (`5B`-like) | mandatory unless repeated full-run evidence already exists |
| blocker characterization (`5C`-like) | mandatory |
| preservation-versus-completeness attribution (`5D`-like) | conditionally mandatory |

### High-Level Blocker-Oriented Branch

Recommended high-level branch structure:

1. blocker entry qualification
   - confirm current corpus, ownership chain, and authoritative artifacts are sufficient for active lifecycle entry
2. blocker persistence validation
   - prove blocker reproducibility across repeated full runs
3. blocker characterization
   - quantify the blocked population, blocker reasons, and opportunity structure
4. blocker attribution
   - determine whether the blocker is evidence-absent, governance-preserving, implementation-driven, or mixed
5. blocker decision point
   - choose among:
     - defer to evidence-acquisition track
     - stop as sufficiently characterized
     - preserve blocked posture pending future pathway work
     - later scorer/evaluator planning track, if separately authorized

### Why Formalization Is Warranted

Formalization is warranted because the current regimen has now been exercised across:

1. a successful aligned surface;
2. an evidence-availability blocker;
3. a stable mixed blocker.

Without a formal blocker-oriented branch, the direct-answer sequence would remain an implicit special case.

The repository evidence now supports making it an explicit regimen option.

Any actual adoption of that branch remains a separate process-governance decision outside this package.

## Determinations

1. Packages `5A` through `5D` changed understanding at every step without changing lifecycle state.
2. The direct-answer surface teaches a reusable blocker-oriented assessment path, not just a one-off exception.
3. The regime now has evidence for at least three branches: alignment, evidence-availability, and blocker-oriented.
4. Packages `5A`, `5B`, and `5C` should be treated as mandatory core steps for future blocker-oriented surfaces.
5. Package `5D` should be treated as conditionally mandatory when governance-preserving missingness and implementation-path concerns may coexist.
6. The direct-answer surface is sufficiently characterized for the current assessment layer and should remain blocked pending future scorer pathway work.
7. A formal blocker-oriented regimen branch is now recommended at the high level, but not adopted by this package.
